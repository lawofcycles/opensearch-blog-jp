コンタクトセンターの AI エージェントを構築したものの、毎回の通話を初めてのように扱ってしまうことに気づいた経験はありませんか。ほとんどのコンタクトセンターシステムでは、通話が終了した瞬間に会話のコンテキストが失われます。会話の中で自然に出てくるライフイベント、好み、関心事といった情報は次のやり取りに引き継がれません。

本記事では、[OpenSearch のエージェントメモリ](https://docs.opensearch.org/latest/ml-commons-plugin/agentic-memory/)を使い、コンタクトセンター AI エージェントに通話をまたいだ永続的な記憶を持たせる方法を紹介します。仕組みの核になるのは、セマンティック検索で必要な記憶だけを引き出す取得機構と、事実(SEMANTIC)、好み(USER_PREFERENCE)、セッション要約(SUMMARY)という 3 種類のメモリ戦略です。これらが同時に動くことで、顧客ごとにパーソナライズされた対応が可能になります。

本記事ではウェルスマネジメント(資産運用)のコンタクトセンターエージェントを例に説明します。ウェルスマネジメントでは、顧客は数カ月から数年にわたる複数の通話の中で退職計画やライフイベント、リスク許容度について話します。1 回の通話では全体像が見えず、必要なコンテキストは顧客の人生の変化に応じて変わっていきます。こうした特性があるため、3 つのメモリ戦略の連携を示すのに適したシナリオです。

## AI エージェントにとってのメモリとは

大規模言語モデル(LLM)はステートレスです。すべての会話が白紙の状態から始まります。AI エージェントがやり取りをまたいで情報を記憶するには、メモリを外部に構築する必要があります。効果的なメモリシステムには 3 つの機能が求められます。会話の中から関連情報を自動的に**保存**すること、必要なタイミングで適切な記憶を**取得**すること、新しい情報に応じて記憶を**更新**することです。内部的な動作の詳細については、[OpenSearch をエージェントメモリソリューションとして使用するブログ記事](https://opensearch.org/blog/opensearch-as-an-agentic-memory-solution-building-context-aware-agents-using-persistent-memory/)を参照してください。

## エージェントメモリに OpenSearch を使う理由

多くのユースケースでは、ベクトル類似度だけで始められます。しかし、コンタクトセンターがスケールすると、情報を取得するクエリにはセマンティックな関連性、キーワードの精度、時間的なフィルタリングを組み合わせる必要が出てきます(例えば「過去 90 日間」の「退職計画に関する会話」をキーワード「ポートフォリオリバランス」で検索する場合)。OpenSearch はネイティブなハイブリッド検索でこれら 3 つの機能をすべてサポートするため、複数のシステムを組み合わせる必要がありません。

また OpenSearch では、短期メモリや長期メモリの構築をエージェント側で実装する必要がありません。短期メモリ(セッションコンテキストや要約)と長期メモリ(事実や好み)は、エージェントメモリの処理戦略としてサーバー側で管理されます。エージェントのコードに抽出やエンベディング管理のロジックを組み込む代わりに、会話を送信するだけで OpenSearch がそれらを処理します。エージェントは会話そのものに集中でき、抽出品質の改善やエンベディングモデルの変更もエージェント側のコードに手を入れずに行えます。

エージェントメモリは API で以下の機能を提供します。

- **[メモリコンテナ](https://docs.opensearch.org/latest/ml-commons-plugin/agentic-memory/#memory-containers)** — 単一コンテナ内の名前空間による顧客ごとの分離
- **[3 つの処理戦略](https://docs.opensearch.org/latest/ml-commons-plugin/agentic-memory/#memory-processing-strategies)** — `SEMANTIC`(事実)、`USER_PREFERENCE`(好み)、`SUMMARY`(セッションコンテキスト)が同時に実行される
- **[自動的な事実抽出](https://docs.opensearch.org/latest/ml-commons-plugin/agentic-memory/#inference-mode)** — `infer: true` を指定して会話を追加すると、構造化された事実、好み、要約が自動抽出される
- **[セマンティック検索](https://docs.opensearch.org/latest/ml-commons-plugin/api/agentic-memory-apis/semantic-search-memory/)** — 関連する記憶を取得する自然言語クエリ

OpenSearch でエージェントメモリをゼロから設定する方法については、[OpenSearch エージェントメモリのブログ記事](https://opensearch.org/blog/opensearch-as-an-agentic-memory-solution-building-context-aware-agents-using-persistent-memory/)を参照してください。

## コンタクトセンターエージェントの構築

ここからは、エージェントメモリをコンタクトセンター AI エージェントに組み込む際のコンポーネントと実装の流れを説明します。

### 前提条件

本記事の例に沿って進めるには、以下のコンポーネントが必要です。

- **OpenSearch 3.6.0 以降** — 本記事で使用するエージェントメモリのセマンティック検索およびハイブリッド検索 API には OpenSearch 3.6.0 が必要です。[Docker](https://docs.opensearch.org/latest/install-and-configure/install-opensearch/docker/) を使用してローカルで OpenSearch を実行できます。
- **登録済み LLM**(事実抽出用)。LLM は通常、ホストされた API(例: Amazon Bedrock、OpenAI、Cohere)への[リモートコネクタ](https://docs.opensearch.org/latest/ml-commons-plugin/remote-models/connectors/)を使用して登録します。
- **登録済みテキストエンベディングモデル**(セマンティック検索用)。Amazon Titan Embeddings、OpenAI、Cohere Embed などの[外部ホストモデル](https://docs.opensearch.org/latest/ml-commons-plugin/remote-models/index/)の使用を推奨します。エンベディング容量が OpenSearch クラスタとは独立してスケールするためです。実験用には[事前学習済みローカルモデル](https://docs.opensearch.org/latest/ml-commons-plugin/pretrained-models/)(例: `all-MiniLM-L12-v2`)も使用できます。

各顧客通話は以下の図に示すライフサイクルに従います。

![](/images/personalizing-your-contact-center-agent-using/628e77ff4159.png)
*Figure 1: 通話ライフサイクル — セッションの 3 つのフェーズ: 挨拶、会話、記憶。*

ライフサイクルは 3 つのフェーズで構成されます。

1. **挨拶**: エージェントが顧客の名前と関連コンテキストをメモリから取得する
2. **会話**: エージェントが会話の必要に応じてオンデマンドで特定の記憶を検索する
3. **記憶**: 通話終了時にエージェントが会話全体をエージェントメモリに保存する

### エージェントメモリの使い方

エージェントメモリはセッション**間**の情報永続化を担います。通話中の会話コンテキストはエージェントフレームワーク自体(LLM のコンテキストウィンドウやセッション状態)で管理されます。通話が終わると、エージェントメモリが重要な情報を保存します。次回以降の通話では、過去のコンテキストをすべて読み込むのではなく、必要に応じてオンデマンドで特定の記憶を取得します。

### 会話をメモリに保存する

エージェントが `infer: true` を指定して会話を送信すると、以下が行われます。

1. OpenSearch が生の会話を**ワーキングメモリ**に未処理のトランスクリプトとして保存する
2. LLM が構造化された知識を抽出し、**長期メモリ**に保存する。この構造化知識には事実、好み、要約が含まれ、それぞれセマンティック検索用のエンベディングを持つ

`infer` が `false`(デフォルト)の場合、OpenSearch はステップ 1 のみを完了します。生の会話をワーキングメモリに保存しますが、構造化知識の抽出は行いません。

クエリ時に、エージェントはセマンティック検索を使用して長期メモリから読み取り、会話に必要な特定の情報(例: 「退職計画」や「リスク許容度」)をクエリします。ワーキングメモリは生の会話ログとして機能し、監査、デバッグ、LLM や戦略を変更した場合の知識再抽出に有用ですが、会話中にエージェントがクエリすることはありません。

### メモリコンテナ内のユーザー分離

すべてのユーザーはメモリコンテナ内の同じ物理ストレージ(ワーキングインデックスと長期インデックス)を共有します。ユーザーの分離はインデックスを分けるのではなく、名前空間(namespace)のタグ付けで実現しています。各メモリには **名前空間**(このケースでは発信者の電話番号を値とする `customer_id`)が付与されます。エージェントがメモリを検索する際はこの名前空間でフィルタリングするため、各顧客は自分の履歴のみを参照できます。以下に構造を示します。

```
Memory Container
  ├── working index (全ユーザーで共有)
  │     ├── messages: [生の会話ターン]
  │     ├── namespace: {customer_id: "+1-555-0123"}  ← ユーザーごとのフィルタ
  │     └── infer: true/false
  │
  └── long-term index (全ユーザーで共有)
        ├── memory: "顧客は保守的な投資を好む"  ← 抽出された知識
        ├── memory_embedding: [...]  ← セマンティック検索用
        ├── strategy_type: SEMANTIC | USER_PREFERENCE | SUMMARY
        └── namespace: {customer_id: "+1-555-0123"}  ← ユーザーごとのフィルタ
```

このエンドツーエンドのプロセスを示すために、音声ベースのウェルスマネジメントコンタクトセンター AI エージェントにエージェントメモリを統合しました。エージェントは Speech-to-Speech LLM を使用して電話に対応していましたが、通話をまたいで顧客を記憶する能力がありませんでした。

エージェントメモリを追加することで、エージェントは以下が可能になりました。

- 3 つのメモリ戦略(`SEMANTIC`、`USER_PREFERENCE`、`SUMMARY`)を同時に使用して、通話をまたいですべての顧客を記憶する
- セマンティック検索とハイブリッド検索を使用して関連する記憶のみを取得する
- カスタムメモリインフラストラクチャなしで、時間の経過とともに各顧客のより豊かなプロフィールを構築する

以下の例では、メモリなしの場合とエージェントメモリありの場合の顧客体験を比較します。

#### メモリなし: 毎回の通話が事前知識なしで始まる

**通話 1 — 退職計画**: 顧客が 5 年後の退職について相談するために電話します。ポートフォリオの詳細、リスクに関する懸念、タイムラインを説明します。エージェントは生産的な会話を行います。エージェント: 「5 年というタイムラインがあれば、段階的にリバランスする余裕があります。奥様と一緒にレビューできるように、いくつかの選択肢を準備しましょうか？」

**通話 2 — 顧客がライフイベントについて再度電話**: 顧客: 「前回話し合ったことについて考えていました。それと、妻と私は小さい家に引っ越すことを考えています。」エージェント: 「お手伝いできますよ。現在の財務状況と目標について教えていただけますか？」

エージェントは「前回話し合ったこと」が何を指すのか分かりません。顧客は退職計画、ポートフォリオ構成、リスク許容度をゼロから説明し直す必要があります。

**通話 3 — 顧客が新しい目標を伝える**: 顧客: 「娘に赤ちゃんが生まれました。孫のために教育資金を設立したいのですが。」エージェント: 「おめでとうございます！教育資金の選択肢をお手伝いできます。現在の財務状況とリスク許容度はいかがですか？」

エージェントは退職計画も住宅売却のことも知らないため、教育資金をこれらと関連付けることができません。

#### エージェントメモリあり: コンテキストが永続化される

**通話 1 — 退職計画**: 顧客が 5 年後の退職について相談するために電話します。大きなポートフォリオを持ち、より保守的な投資を望んでいます。AI エージェントが選択肢を議論し、会話(顧客のリスクに関する懸念やタイムラインを含む)が自動的に抽出・保存されます。

**通話 2 — ライフイベント**: 同じ顧客が 1 カ月後に電話します。顧客: 「前回話し合ったことについて考えていました。それと、妻と私は小さい家に引っ越すことを考えています。」エージェント: 「お帰りなさい！前回は、5 年後の退職に向けてポートフォリオをより保守的な投資にシフトすることについて話しましたね。現在のお住まいを売却すれば相当な資金が生まれます。その収益をリバランスのタイムライン加速にどう活用できるか、モデリングしてみましょうか？奥様と一緒にレビューできるように選択肢を準備しますよ。」

**通話 3 — 新しい目標**: 顧客: 「娘に赤ちゃんが生まれました。孫のために教育資金を設立したいのですが。」エージェント: 「おめでとうございます！退職計画と以前お話しした住宅売却の可能性を考えると、いくらか柔軟性があります。全体的な計画を補完する保守的な教育資金が合理的でしょう。リバランス戦略を変更する必要もありません。奥様と一緒にレビューできるように、いくつかの選択肢をまとめましょうか？」

顧客が「教育資金」に言及したとき、エージェントはそれを退職計画や住宅売却といった顧客の財務全体像の中に位置づけて理解できます。

メモリがなければ、すべてのやり取りはゼロから始まります。顧客は同じことを繰り返し、エージェントにはコンテキストがなく、関係性を深める機会が失われます。顧客が家の売却に言及しても、退職計画を知らなければ、エージェントは両者の関係を理解できません。孫が生まれても、財務全体像がなければ、エージェントは個別にカスタマイズされたアドバイスではなく一般的なアドバイスしか提供できません。

メモリがあれば、エージェントは前回の会話の続きから始められます。顧客は理解されていると感じます。エージェントはコンテキスト収集よりもアドバイスに多くの時間を費やせます。ライフイベントが新しい財務ニーズを生んだとき、エージェントはそれを認識します。全体像を記憶しているからです。

この違いは効率だけの話ではありません。単に問い合わせを処理するサービスと、顧客との関係を積み重ねていくサービスとの違いです。

## アーキテクチャ概要

以下のアーキテクチャは特定のコンポーネントに縛られない設計になっています。音声エージェント、チャットエージェント、API 駆動のエージェントのどれであっても、メモリレイヤーの部分は同じ構成で動きます。私たちの実装では Speech-to-Speech LLM を使ったコンタクトセンターを採用しましたが、OpenSearch 側のコンポーネント(エージェントメモリ、LLM、エンベディングモデル)はエージェントの種類によらず共通です。

![](/images/personalizing-your-contact-center-agent-using/088cc253b060.png)
*Figure 2: アーキテクチャ概要 — AI エージェントが S2S LLM、OpenSearch エージェントメモリ、メモリで使用される LLM + エンベディングモデルをオーケストレーションする。*

アーキテクチャは以下のコンポーネントで構成されます。

- **コンタクトセンター** — 通話を受信し、テレフォニーを AI エージェントに接続する。クラウドベースでもオンプレミスでも、メモリレイヤーの動作は同じ。
- **AI エージェント** — 会話をオーケストレーションする。通話中にオンデマンドでメモリを検索し、通話終了時にメモリを書き込み、ビジネスロジックを適用し、プロンプトを構築する。
- **Speech-to-Speech LLM** — エージェントの音声インターフェース。音声入出力を処理する。
- **OpenSearch エージェントメモリ** — 永続的なメモリレイヤー。本記事の焦点。
- **LLM + エンベディングモデル** — エージェントメモリ内での事実抽出(推論による)とセマンティック検索を実行する。

## 詳解: 3 つのメモリ戦略の連携

OpenSearch エージェントメモリは、単一のメモリコンテナ内で同時に実行される 3 つの[メモリ戦略](https://docs.opensearch.org/latest/ml-commons-plugin/agentic-memory/#memory-processing-strategies)をサポートします。それぞれ会話の異なる側面を捉えます。

- **SEMANTIC** — 事実やライフイベントを捉える。退職目標、ポートフォリオ構成、家族の変化など。
- **USER_PREFERENCE** — 顧客がどのようにサービスを受けたいかを捉える。リスク許容度、コミュニケーションスタイル、意思決定の好みなど。
- **SUMMARY** — 凝縮されたセッションコンテキストを捉える。何が議論され、何が決定され、何が保留中かなど。

メモリコンテナの作成時に 3 つのメモリ戦略すべてを設定します。各顧客のメモリは名前空間で分離されるため、顧客ごとに別のコンテナを用意する必要はありません。3 つの戦略すべてを含むコンテナの例を以下に示します。

```json
POST /_plugins/_ml/memory_containers/_create
{
  "name": "wealth-advisor",
  "configuration": {
    "embedding_model_type": "TEXT_EMBEDDING",
    "embedding_model_id": "your-embedding-model-id",
    "embedding_dimension": 1024,
    "llm_id": "your-llm-model-id",
    "strategies": [
      { "type": "SEMANTIC", "namespace": ["customer_id"] },
      { "type": "USER_PREFERENCE", "namespace": ["customer_id"] },
      { "type": "SUMMARY", "namespace": ["customer_id"] }
    ]
  }
}
```

コンテナ作成 API とコンフィグオプションの詳細については、[エージェントメモリのドキュメント](https://docs.opensearch.org/latest/ml-commons-plugin/agentic-memory/#memory-containers)を参照してください。

### 戦略の抽出カスタマイズ

各戦略はコンフィグでオプションの `system_prompt` を受け付け、何を抽出するかを制御できます。ウェルスマネジメントエージェントの場合、SEMANTIC を財務的事実(退職タイムライン、ポートフォリオ構成、計画に影響するライフイベント)に絞り、本題と無関係な雑談を無視するように設定できます。USER_PREFERENCE は、「メールを好む」のような表面的な好みではなく、意思決定スタイル(例: 配偶者の関与、リスク許容度)を捉えるように調整できます。これらのカスタムプロンプトは汎用性と引き換えに精度を高めており、ドメインに明確な語彙がある場合に効果を発揮します。

### 連携の仕組み: ウェルスマネジメントのフロー

以下の例は、複数の顧客通話にわたって 3 つのメモリ戦略がどのように連携するかを示します。

**通話 1 — 退職計画**:

顧客が退職について電話で相談します。通話終了時に、エージェントは会話全体をエージェントメモリに送信します。

```json
POST /_plugins/_ml/memory_containers/<container_id>/memories
{
  "messages": [
    {
      "role": "user",
      "content": [{ "text": "5 年後くらいに退職しようと考えています。ポートフォリオのほとんどが株式で、ボラティリティが心配になってきました。もっと保守的なものにシフトしたいのですが。", "type": "text" }]
    },
    {
      "role": "assistant",
      "content": [{ "text": "そのタイムラインなら理にかなっていますね。債券や配当株のミックスに向けて段階的にリバランスできます。奥様と一緒にレビューできるように、いくつかの選択肢を準備しましょうか？", "type": "text" }]
    },
    {
      "role": "user",
      "content": [{ "text": "はい、お願いします。妻は決める前に数字を見たいと思いますので。", "type": "text" }]
    }
  ],
  "payload_type": "conversational",
  "infer": true,
  "namespace": { "customer_id": "+1-555-0123" },
  "tags": { "topic": "retirement_planning" }
}
```

`infer` が `true` の場合、LLM は自動的に事実(`SEMANTIC`: 5 年後の退職、株式中心のポートフォリオ)を抽出し、好み(`USER_PREFERENCE`: 保守的、意思決定に配偶者が関与)を検出し、セッションの要約(`SUMMARY`)を生成します。抽出された知識は長期メモリに保存され、セマンティック検索で検索可能になります。`infer: true` なし(デフォルトは `false`)の場合、メッセージは生データとしてのみ保存されます。これはログ記録には有用ですが、意味による検索はできません。会話全体を個別のターンではなく一度に送信することで、LLM は完全なコンテキストを持ち、より豊かで一貫性のある事実と好みを抽出できます。

### 通話終了時に保存する理由

LLM は会話全体を一度に見た方がより良い抽出結果を生み出します。あるターンの「5 年後に退職」と別のターンの「妻が数字を見たがっている」を結びつけ、共同意思決定の好みを推論できます。ターンごとに保存すると、これらは孤立した事実として抽出されてしまい(「顧客は退職を計画している」「顧客の妻は数字を見たがっている」)、両者の関係が失われます。断片化された事実は取得時にノイズを加えるだけで、重要なインサイトを捉えられません。

**通話 2 — ライフイベント**:

同じ顧客が 1 カ月後に電話します。コンタクトフローが顧客の名前をメモリから検索し、パーソナライズされた挨拶を生成します。名前の検索は同じセマンティック検索 API を使用し、`k` は返す最大結果数を指定します。

```json
POST /_plugins/_ml/memory_containers/<container_id>/memories/long-term/_semantic_search
{
  "query": "customer name",
  "k": 3,
  "namespace": { "customer_id": "+1-555-0123" }
}
```

その後、会話が始まります。顧客が「前回話し合ったこと」に言及すると、エージェントは LLM に現在の会話ターンに基づいて取得クエリを生成させます。LLM は「retirement planning portfolio rebalancing」のような具体的で的を絞ったクエリを生成するかもしれません。

```json
POST /_plugins/_ml/memory_containers/<container_id>/memories/long-term/_semantic_search
{
  "query": "retirement planning portfolio rebalancing",
  "k": 10,
  "namespace": { "customer_id": "+1-555-0123" }
}
```

エージェントは退職タイムライン、リスク許容度、配偶者が関与していることを見つけ、すぐに引っ越しを既存の計画に結びつけられます。「現在のお住まいを売却すれば相当な資金が生まれます。その収益をリバランスのタイムライン加速にどう活用できるか、モデリングしてみましょうか？」

**通話 3 — 新しい目標**:

顧客が孫のための教育資金に言及します。この新しいトピックに対して、エージェントは LLM に顧客のより広い財務コンテキストを提供する取得クエリを生成させます。LLM は「financial situation retirement savings」のようなクエリを生成するかもしれません。

```json
POST /_plugins/_ml/memory_containers/<container_id>/memories/long-term/_semantic_search
{
  "query": "financial situation retirement savings",
  "k": 10,
  "namespace": { "customer_id": "+1-555-0123" }
}
```

完全なコンテキストを取得した上で、エージェントは以下のように応答します。

「おめでとうございます！約 5 年後の退職と住宅売却の可能性を考えると、いくらか柔軟性があります。全体的な計画を補完する保守的な教育資金が合理的でしょう。奥様と一緒にレビューできるように選択肢を準備しましょうか？」

エージェントは退職のタイムライン、住宅売却の議論、保守的な好み、配偶者が関与したいことをすべて前回の通話から記憶していました。

## セマンティック検索とハイブリッド検索: 必要な情報を見つける

OpenSearch にセマンティック検索が導入される前は、メモリの取得には事前生成されたエンベディングを使った k 近傍法(k-NN)クエリを手動で構築する必要がありました。[セマンティック検索](https://docs.opensearch.org/latest/ml-commons-plugin/api/agentic-memory-apis/semantic-search-memory/)と[ハイブリッド検索](https://docs.opensearch.org/latest/ml-commons-plugin/api/agentic-memory-apis/hybrid-search-memory/) API は、メモリ取得を大幅に簡素化します。

### セマンティック検索

セマンティック検索では、自然言語クエリを送信すると意味的に関連するメモリを受け取れます。OpenSearch がエンベディング生成とベクトル検索を自動的に処理します。

```json
POST /_plugins/_ml/memory_containers/<container_id>/memories/long-term/_semantic_search
{
  "query": "client risk tolerance and investment preferences",
  "k": 5,
  "namespace": { "customer_id": "+1-555-0123" },
  "min_score": 0.6
}
```

このクエリは、保存された会話にその正確な言葉が一度も出てこなくても、顧客の保守的なスタンスや段階的な変更への好みに関するメモリを返します。典型的な応答は以下のようになります。

```json
{
  "hits": {
    "hits": [
      {
        "_source": {
          "memory": "Client prefers conservative investments and wants to shift away from equities",
          "strategy_type": "USER_PREFERENCE",
          "namespace": { "customer_id": "+1-555-0123" }
        }
      },
      {
        "_source": {
          "memory": "Client plans to retire in five years with a gradual rebalancing strategy",
          "strategy_type": "SEMANTIC",
          "namespace": { "customer_id": "+1-555-0123" }
        }
      }
    ]
  }
}
```

各結果には、抽出された `memory` テキスト、それを生成した `strategy_type`、所属する `namespace` が含まれます。エージェントは `memory` フィールドを読み取り、関連する事実をプロンプトに注入します。

私たちの実装では、エージェントは会話中にオンデマンドでセマンティック検索を使用します。顧客が「私の退職目標は何でしたか？」と聞くと、エージェントは `query="retirement savings target"` でメモリを検索し、関連する結果のみを取得します。会話が投資の好みに移ると、エージェントは `query="investment risk tolerance"` で検索します。これにより各取得が焦点を絞られ、無関係なコンテキストを LLM プロンプトに読み込むことを防ぎます。

### 最初にすべてのメモリを読み込むのではなくオンデマンド検索を行う理由

より単純なアプローチは、各通話の開始時に顧客のすべてのメモリを読み込み、LLM プロンプトに注入することです。最初はうまく機能しますが、メモリがセッションを重ねて蓄積されると 2 つの問題が生じます。プロンプトが無関係なコンテキストで過負荷になることと、固定的なクエリでは会話が何を必要とするか予測できないことです。

オンデマンド検索は両方の問題を解決します。エージェントは現在のターンに関連するものだけを取得し、クエリは実際の会話によって形成されます。

### 検索時の戦略の機能

3 つのメモリ戦略(`SEMANTIC`、`USER_PREFERENCE`、`SUMMARY`)は書き込み時にそれぞれ異なる抽出パイプラインとして動作します。一方、検索時にはセマンティック検索が単一のインデックスに対してまとめてクエリを実行します。戦略ごとのフィルタはかからず、結果はエンベディングの類似度だけでランク付けされます。

この仕組みにより、**クエリの内容に応じて最も適した戦略のメモリが自然に上位に来る**という振る舞いが生まれます。実際の会話データでテストすると、以下のパターンが確認できます。

- 「what are my preferences」→ 上位結果: リスク許容度、コミュニケーションスタイル、意思決定の好み。支配的な戦略: `USER_PREFERENCE`
- 「summarize our past conversations」→ 上位結果: セッション要約、会話フローのメモ。支配的な戦略: `SUMMARY`
- 「retirement age and savings」→ 上位結果: 年齢、退職日、貯蓄額。支配的な戦略: `SEMANTIC`(一部 `USER_PREFERENCE` も含む)

これは意図的に設計されたルーティングではなく、各戦略が生成するテキストの語彙的な特性から自然に生じる結果です。`USER_PREFERENCE` のメモリは好みに関する表現を多く含み、`SUMMARY` のメモリは要約調の表現を含むため、セマンティック検索がそれぞれ対応するクエリに適切にマッチします。

つまり、戦略を意識した取得ロジックをアプリケーション側で実装する必要はありません。1 回のセマンティック検索クエリが、適切な戦略の結果を混ぜて返してくれます。

ウェルスマネジメントの例で言えば、通話冒頭で「前回話し合ったこと」のような広い話題が出た場合、セマンティック検索は `SUMMARY` メモリ(前回の退職計画に関するセッション全体の要約)と `SEMANTIC` な事実を一緒に返す傾向があります。「retirement planning portfolio rebalancing」のようにもっと具体的なクエリでは `SEMANTIC` な事実が中心になり、顧客の希望するサービス提供スタイルに関する質問では `USER_PREFERENCE` が主に返されます。

書き込み時に 3 つの戦略をすべて実行しておく利点は、将来の会話でどの切り口の情報が必要になるか予測しなくてよい点です。LLM がどんなクエリを生成しても、対応するメモリ表現がすでに準備されています。

### ハイブリッド検索

キーワードの精度とセマンティックな理解の両方が有益なクエリに対して、エージェントメモリは BM25 キーワードマッチングとニューラル検索を組み合わせた `_hybrid_search` エンドポイントも提供します。私たちの実装ではセマンティック検索で十分でした。しかし、セマンティックな関連性とともに正確なキーワードマッチングが必要な場合にハイブリッド検索を利用できます。

```json
POST /_plugins/_ml/memory_containers/<container_id>/memories/long-term/_hybrid_search
{
  "query": "retirement timeline five years portfolio rebalancing",
  "k": 10,
  "namespace": { "customer_id": "+1-555-0123" }
}
```

このクエリは、言及された正確なタイムライン(キーワードマッチ)と関連するファイナンシャルプランニングのコンテキスト(セマンティックマッチ)の両方を見つけます。

**注**: `_hybrid_search` エンドポイントは `_semantic_search` より新しいです。完全な API 仕様については、[ハイブリッド検索 API リファレンス](https://docs.opensearch.org/latest/ml-commons-plugin/api/agentic-memory-apis/hybrid-search-memory/)を参照してください。ハイブリッド検索 API は BM25 キーワードスコアとエンベディング類似度スコアの組み合わせでランク付けされた結果を返します。

## エージェントメモリのベストプラクティス

以下のヒントは、このシステムの構築とテストに基づいています。

- **名前空間を早い段階で設計する**: すべてのセッションの開始時にユーザー入力なしで利用できる名前空間キーを選択してください。私たちはテレフォニーシステムが自動的に提供する発信者の電話番号を使用しました。チャットエージェントの場合、認証済みユーザー ID が適しています。重要な要件は、セッション間で安定しており、会話が始まる前に利用可能であることです。
- **事実抽出に高性能な LLM を選択する**: `USER_PREFERENCE` 戦略はカテゴリとコンテキスト注釈を含む構造化 JSON を生成するため、整形式の JSON を安定して生成できるモデルが必要です。Anthropic Claude Sonnet のような高性能モデルで最良の結果が得られましたが、抽出の複雑さによっては小規模モデルでも動作する可能性があります。各戦略を選択した LLM でテストし、`strategy_type` フィールドごとの長期メモリ数を監視して、すべての戦略が出力を生成していることを確認してください。
- **具体的なクエリを設計する**: オンデマンド取得では、検索品質はクエリに依存します。「customer information」のようなクエリは広範な事実の混合を返しますが、「retirement timeline」や「investment risk tolerance」のようなクエリはより的を絞った結果を返します。エージェントは会話コンテキストからクエリを構築するため、プロンプトは具体性を促進する必要があります。
- **適切なエンベディングモデルを選択する**: エージェントメモリは事前学習済みローカルモデルとリモートコネクタの両方をエンベディングに対応しています。本番環境では、リモートモデル(例: Amazon Titan Embeddings、OpenAI、Cohere Embed)によりエンベディング容量が OpenSearch クラスタとは独立してスケールし、再デプロイなしでモデルのアップグレードが可能です。ローカルの事前学習済みモデル(例: `all-MiniLM-L12-v2`)は実験に有用ですが、本番使用前にセマンティック検索の品質を検証してください。
- **グレースフルなフォールバック処理を追加する**: メモリ操作をエラーハンドリングで包むことで、OpenSearch が一時的に利用不可能な場合でもエージェントの応答性を維持できます。この場合、エージェントはメモリアクセスが復旧するまでステートレスな会話にフォールバックします。
- **必要に応じてセマンティック検索と新しさを組み合わせる**: 最新情報が重要なクエリ(例: 「現在のポートフォリオ配分」)では、セマンティック結果と `created_time` による最新 N 件のメモリを組み合わせて、最近の更新を見逃さないようにしてください。
- **ユースケースに合わせて取得パラメータを調整する**: `k` の値は取得に直接影響します。的を絞ったクエリでは、中程度の `k`(5〜10)で通常十分です。クエリの具体性は `k` よりも大きな影響を与えます。的を絞ったクエリで `k=5` の方が、一般的なクエリで `k=50` よりも優れた結果を出すことが多いです。本番環境で取得品質を監視し、メモリ量の増加に応じて調整してください。

## まとめ

通話のたびに顧客を忘れるコンタクトセンター AI エージェントは、非効率なだけでなく、本来得られるはずの信頼関係を築く機会を逃しています。アドバイザーの対応力を高める情報(ライフイベント、好み、過去の会話履歴)は、普段の会話の中にすでに含まれています。課題は、その情報を体系的に蓄積し、必要なときに引き出せるようにすることです。

OpenSearch エージェントメモリは、この課題を一連の API 呼び出しで解決します。3 つの戦略が並行して動き、同じ会話から事実を抽出し、好みを学習し、セッションの要約を生成します。有効化に必要なのは `infer: true` フラグだけです。セマンティック検索を使えば、蓄積されたメモリの中から今の会話に必要なものだけを取り出せます。

私たちはウェルスマネジメント向けにこのシステムを構築しました。顧客との関係が何年にもわたり、やり取りを重ねるごとにコンテキストが積み上がっていくドメインです。同じパターンは、保険金請求、医療フォローアップ、テクニカルサポートのエスカレーションなど、顧客が「前回の続き」を期待するあらゆる場面に適用できます。
